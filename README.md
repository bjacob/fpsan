# fpsan — a stand-alone float32 toy

A header-only C++ toy that reproduces the core of Triton's **FPSan**
floating-point sanitizer in plain `float32`, so you can play with
FPSan-rewritten arithmetic by hand without building Triton.

FPSan isn't ours — it was invented by the [Triton
project](https://github.com/triton-lang/triton), and the ideas studied here (the
embedding into modular integers, the algebraic semantics, the link to Schanuel's
conjecture) are the FPSan authors'. The math is explained in their blog post,
["Schanuel's conjecture and the semantics of
FPSan"](https://cp4space.hatsya.com/2026/05/03/schanuels-conjecture-and-the-semantics-of-fpsan/);
see also the [Triton FPSan
docs](https://github.com/triton-lang/triton/blob/main/docs/programming-guide/chapter-3/fpsan.rst).
This repo ports their constants verbatim from
[`FpSanToLLVM.cpp`](https://github.com/triton-lang/triton/blob/main/lib/Conversion/TritonInstrumentToLLVM/FpSanToLLVM.cpp)
and
[`FpSanitizer.cpp`](https://github.com/triton-lang/triton/blob/main/lib/Dialect/TritonInstrument/Transforms/FpSanitizer.cpp),
for study rather than production.

For a real, complete implementation — every GPU intrinsic, mixed precision, a
full test suite — see **[ROCm/hip-fpsan](https://github.com/ROCm/hip-fpsan)**.

**Start here:** [understanding-fpsan.md](understanding-fpsan.md) derives the
whole construction from first principles, one ingredient at a time;
[narrow-floats.md](narrow-floats.md) works the same recipe by hand on 4- and
8-bit floats, small enough to check every embedding table by eye.

- `fpsan_f32.hpp` — the `FPSanF32` class and the embedding `phi`. **This is the
  primary code**: one concrete type, nothing to configure.
- `demo.cpp` — toy examples confirming the properties the blog post claims.

Requires **C++20** (uses `std::bit_cast`). Just invoke your compiler directly:

```
c++ -std=c++20 -O2 demo.cpp -o demo && ./demo
```

## What it is

FPSan replaces floating-point arithmetic with **integer arithmetic modulo
2³²**. The numerical results are scrambled and useless for computation — that is
deliberate. The point is that the integer image obeys *exact* algebraic laws
(associativity, commutativity, distributivity, and an exponential homomorphism
`exp(x+y) = exp(x)·exp(y)`) that real IEEE arithmetic only obeys approximately.
So two programs that differ only by "fast-math" rewrites such as
`a+(b+c) → (a+b)+c` — invalid over floats, since FP addition is not associative —
become *bit-for-bit identical* once rewritten by FPSan, because the rewrite is
exact on modular integers.

Under the (believed) real version of Schanuel's conjecture, the subring of ℝ
generated by `{0, 1, +, ×, exp}` is the *free* exponential ring, and the
integer model `(ℤ/2³², +, ×, C^·)` is a quotient of it — which is why
algebraically-equivalent circuits stay equivalent through the rewrite. (Again:
this is the FPSan authors' insight, explained in their blog post linked above.)

## The construction

`phi : float32 ↔ ℤ/2³²` is a bijection (`embed`/`unembed`) with stable fixed
points `phi(+0.0)=0`, `phi(+1.0)=1`, `phi(-1.0)=0xFFFFFFFF`, and `phi(-x)=-phi(x)`
for nonzero `x`. Floating-point operations become operations on the integer
*payload*:

| FP op       | FPSan payload rewrite                       |
|-------------|---------------------------------------------|
| `x + y`     | `phi⁻¹( phi(x) + phi(y) )`                   |
| `x - y`     | `phi⁻¹( phi(x) - phi(y) )`                   |
| `-x`        | `phi⁻¹( -phi(x) )`                           |
| `x * y`     | `phi⁻¹( phi(x) * phi(y) )`                   |
| `x / y`     | `phi⁻¹( phi(x) * inv(phi(y)) )`             |
| `exp2(x)`   | `phi⁻¹( C ^ phi(x) )`, `C = 0xA343836D` (≡5 mod 8) |
| `exp(x)`    | `exp2` of `phi(x)` rescaled by `0x236EE9BF` (≈ 1/ln 2) |

All arithmetic on payloads is ordinary `uint32_t` wraparound (mod 2³²).

The exact mixing/operation constants are taken verbatim from the Triton sources
(linked at the top of this file), so the payloads match what Triton's FPSan
produces.

## Caveats (matching FPSan's own docs)

- This is **not** an IEEE simulator: no real ordering, rounding, NaN/Inf,
  subnormals, or exceptions.
- `x / x == 1` holds only for **odd** payloads; for even payloads `inv` is a
  parity-preserving involution, not a true inverse (the demo shows one of each).
- FPSan results are only meaningful when compared against *other* FPSan results.

## The narrow-float study material

The code behind [narrow-floats.md](narrow-floats.md). `fpsan_f32.hpp` stays the
primary code; these are supporting study files:

- `fpsan_generic.hpp` — a small, run-time-parameterized companion (`FPFormat` +
  `FPSanFloat`) that applies the same recipe to any narrow format. Deliberately
  plain, not a template framework; it exists to produce the document's tables.
- `narrow_demo.cpp` — enumerates FP4 and FP8 exhaustively, prints every table in
  the document, and verifies the properties (it also cross-checks that the
  generic path reproduces `fpsan_f32.hpp`'s payloads bit-for-bit):

  ```
  c++ -std=c++20 -O2 narrow_demo.cpp -o narrow_demo && ./narrow_demo
  ```
